# Codebase Analysis & Scenario Re-Verification — 2026-06-12

> Historical pre-0.3.0 audit. Several findings below were corrected in 0.3.0; use
> `docs/VALIDATION.md` and the current test suite for present behavior.

Multi-lens codebase analysis plus a full re-test of every feature by scenario. Analysis was
run as a 4-dimension adversarial workflow (science, architecture, code quality, tests): each
dimension's findings were independently verified by a separate agent (default to "refuted"
unless evidence held), then synthesized. 25 agents, 20 findings raised, **19 confirmed,
1 refuted**.

## Verdict

The workbench is scientifically sound at its core. Every constraint-based primitive
re-verified numerically against COBRApy on `e_coli_core`, the architecture invariants hold
(zero Qt leakage into `core`/`features`/`omics`/`visualization`), and the test suite asserts
real scientific values rather than just "runs without error". The risk was concentrated in
(1) unverified GUI paths and (2) an overstated capability surface — both addressed below.

## Confirmed findings deferred (documented, not changed)

Recorded for a future, separately-validated pass; none block current use:

- **science (med):** `mta_miqp` mixes a squared-flux steady penalty with a unit
  transformation count at a hard-coded `weight=1.0`, so its objective trade-off is
  flux-scale-dependent and the public `alpha` parameter does not reach the MIQP. The MIQP is
  a secondary high-fidelity mode and is tested (ranks the known target #1); rescaling its
  objective is a scientific tuning decision out of scope here. (`revert.py`)
- **science (low):** the carbon ceiling assumes the detected substrate is the sole carbon
  source; a multi-carbon medium could exceed it for non-CO₂ reasons. Correct for the typical
  minimal-medium use the GUI drives. (`production.py`)
- **architecture (low):** `eflux2` silently degrades QP→pFBA (L2→L1) when no QP solver is
  present, unlike sibling services that raise `SolverCapabilityError`; the chosen formulation
  is exposed only via a `detail` string. Deliberate graceful degradation; documenting the
  pattern would make it explicit. (`expression.py`)
- **architecture (low):** the solver capability matrix is a hand-maintained dict keyed by
  solver name rather than probed from optlang; the architecture narrative also says "MILP
  needs a commercial solver" while the matrix grants GLPK/HiGHS MILP. Gating is correct; the
  prose and the static table are the drift risk. (`solvers.py`, `architecture.md`)
- **tests (med/low):** the A→B transformation finder and multi-condition log2 fold-change are
  implemented and unit-tested as **services** but are not wired into a GUI tab, while the
  README presents them among the feature set. Either add GUI tabs (with UI tests) or scope
  the README claims to the Python API.
- **tests (low):** several GUI production handlers (`run_fvseof_plot`,
  `run_theoretical_yield` success path, `render_flux_map`) are asserted only on label
  substrings, so a wiring/formatting bug (e.g. an ignored aerobic flag) would pass. The
  underlying services are value-tested; the new scenario tests now at least execute these
  paths end-to-end.

## Refuted finding (verification caught it)

"`theoretical_yield` divides without an optimal-status check, so an infeasible model's NaN
poisons downstream FSEOF/FVSEOF." **Refuted empirically:** with cobra 0.31.1 + Gurobi an
infeasible model returns `status='infeasible'` with 0.0 fluxes (not NaN), so `molar_yield`
is 0.0 and the GUI guard fires correctly. The only real NaN path was the closed-substrate
case, fixed separately above.
